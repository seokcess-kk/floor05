# Part 1. floor05 플랫폼

---

## 1. 프로젝트 개요

### 1.1 배경

floor05(플로어공오)는 "존재하지 않는 0.5층"을 컨셉으로 한 개인 사이드 비즈니스로, 무언가가 필요한데 이게 있을까 싶은 것들을 찾는 모든 사람들을 대상으로 하는 니치 유틸리티 툴 플랫폼이다.

첫 번째 서비스로 이미지 도구 모음을 선정했으며, 브라우저 기반 100% 로컬 처리로 서버 비용 제로의 운영을 목표로 한다.

### 1.2 타겟 사용자

무언가가 필요한데 이게 있을까 싶은 것들을 찾는 모든 사람들. 주로 구글 키워드 검색으로 개별 도구에 직접 진입하는 패턴.

### 1.3 수익 모델

- Google AdSense + Kakao Adfit 광고 수익
- 목표: 월 10,000 PV 이상부터 수익화 시작
- 운영비: $0/월 (브라우저 처리, 서버 비용 제로)
- 도메인: $10~15/년

## 2. 플랫폼 구조

### 2.1 전체 설계 원칙

- 모든 서비스는 floor05.com 하위 경로로 통합 운영 (도메인 권위 누적)
- 사용자는 구글 검색으로 개별 도구에 직접 진입하는 패턴이 주력
- 메인 페이지는 브랜드 소개 + 서비스 발견 역할 (혼합형)
- 서비스 발견은 각 도구 페이지 내부에서 분산 처리

### 2.2 메인 페이지 (floor05.com)

#### 상단: Hero 섹션

브랜드 스토리 모션 그래픽 (CSS/Lottie 애니메이션). "존재하지 않는 0.5층"을 시각적으로 표현하는 비주류 무드의 영상. 자동재생/무음. 브랜드 카피 함께 배치.

#### 중단: 출시 도구 목록

현재 출시된 서비스를 카드 형태로 나열. 각 카드에 인터랙티브 요소 (호버 애니메이션 등). 실제 출시된 서비스만 표시하며 "준비 중" 표시는 하지 않는다.

#### 하단: 브랜드 철학

floor05의 철학을 담은 짧은 텍스트.

**광고: 메인 페이지에는 광고를 배치하지 않는다.**

### 2.3 헤더 네비게이션

| 위치 | 요소 |
|------|------|
| 왼쪽 | floor05 로고 (홈 링크) |
| 오른쪽 | "도구" 드롭다운 \| "블로그" 링크 |

- "도구" 드롭다운: 출시된 모든 서비스를 플랫 리스트로 표시
- 서비스 10개 초과 시 드롭다운 내부에서 카테고리 구분 추가
- 출시되지 않은 서비스는 표시하지 않음

### 2.4 서비스 확장 원칙

| 시점 | 상태 | 액션 |
|------|------|------|
| 초기 (1~9개) | 플랫 리스트, 카테고리 없음 | 헤더 드롭다운에 전체 나열 |
| 성장기 (10개+) | 카테고리 구분 필요 | 드롭다운 내 탭/소제목 추가 |
| 확장기 (10개+) | 카테고리별 허브 필요 | /tools/image 허브 페이지 생성 |

- 어드민 페이지: 현 단계 불필요. 퀴즈/테스트 콘텐츠 시점에 재논의
- URL 구조는 처음부터 `/tools/image/compress` 형태로 설계하여 허브 추가 시 기존 URL 보전

## 3. 공통 설계

### 3.1 도구 페이지 공통 레이아웃

구글 검색으로 직접 진입하는 사용자가 가장 많이 보는 화면. 스크롤 흐름: 도구 사용 → 광고 → 학습 → 다음 행동 유도 → 발견

| # | 영역 | 상세 |
|---|------|------|
| 1 | 헤더 | floor05 로고 + 도구 드롭다운 + 블로그. 얇은 높이, 브랜드 다크 톤 |
| 2 | 타이틀 영역 | 도구명 + 한 줄 설명 (차별화 포인트 포함) |
| 3 | 도구 본체 | 페이지 핵심. 드롭존 → 업로드 → 처리 옵션 → 비교 → 다운로드 |
| 4 | 광고 슬롯 1 | 도구 본체 바로 아래. 다운로드 후 시선 이동 위치 |
| 5 | 사용 가이드 | 3~4단락 SEO 텍스트. FAQ 형태로 구성 (GEO 최적화) |
| 6 | 워크플로우 CTA | "압축 끝났나요?" + 관련 도구 카드 2~3장 (도구 간 연결) |
| 7 | 광고 슬롯 2 | CTA 아래, 다른 도구 섹션 위 |
| 8 | floor05 다른 도구 | 출시된 전체 도구 카드 나열. 포털 역할 분산 |
| 9 | 광고 슬롯 3 | 푸터 위. 페이지 길이에 따라 생략 가능 |
| 10 | 푸터 | floor05 로고 + 서비스 링크 + 저작권 |

### 3.2 디자인 시스템

**공통 뼈대(ToolLayout)는 일관되게, 도구 본체는 자유롭게.**

- ToolLayout이 헤더, 푸터, 광고 슬롯, 타이틀, 다른 도구 섹션을 공통 제공
- 도구 본체 영역은 각 서비스 성격에 맞게 자유 설계
- 컬러와 타이포그래피는 브랜드 가이드 준수
- "각 층마다 다른 분위기의 공간" — floor05 브랜드 컨셉과 일치

### 3.3 광고 배치 원칙

| 원칙 | 상세 |
|------|------|
| 메인 페이지 무광고 | 브랜드 첫인상에 집중 |
| 한 페이지 최대 3개 | 페이지 길이에 따라 2개로 축소 가능 |
| 도구 사용 영역 내 광고 금지 | UX 보호 최우선 |

#### 페이지 유형별

- **도구 페이지**: 도구 아래 1개 + CTA 아래 1개 + 푸터 위 1개
- **블로그**: 인-콘텐츠 2~3개 (글 길이 비례)
- **퀴즈/테스트 (향후)**: 결과 페이지 집중, 진행 중 광고 없음

## 4. SEO & GEO 전략

### 4.1 SEO title/description 패턴

title에 floor05 브랜드명은 넣지 않는다. 키워드 + 차별화 한 마디로 클릭률 극대화.

| 요소 | 패턴 |
|------|------|
| title | [핵심 키워드] - [차별화 한 마디] |
| description | [개인정보 보호 강조] + [핵심 기능 나열] |

**예시 (압축 도구):**
- title: `이미지 압축 - 서버 전송 없이 브라우저에서 바로`
- description: `파일이 내 기기를 떠나지 않습니다. 회원가입 없이 무제한 무료. 목표 용량 설정, 일괄 압축, Before/After 비교까지.`

### 4.2 페이지별 SEO 키워드

| 페이지 | 타겟 키워드 |
|--------|------------|
| /tools/image/compress | 이미지 용량 줄이기, 사진 압축, 이미지 압축 |
| /tools/image/resize | 사진 크기 줄이기, 이미지 리사이즈, 인스타 사진 크기 |
| /tools/image/convert | PNG JPG 변환, 이미지 변환, WebP 변환 |
| /tools/image/heic-to-jpg | 아이폰 사진 변환, HEIC JPG 변환 |
| /tools/image/crop | 이미지 자르기, 사진 자르기 |

### 4.3 내부 링크 구조

- **메인 → 각 도구**: 도구 카드에서 링크
- **도구 → 도구**: 워크플로우 CTA + "다른 도구" 섹션
- **블로그 → 도구**: 가이드 글 내 자연스러운 CTA
- **도구 → 블로그**: 가이드 섹션에서 "더 자세한 내용" 링크

### 4.4 GEO (Generative Engine Optimization)

AI 검색 (Google AI Overview, Bing Copilot, Perplexity)에서 floor05가 인용/추천되도록 최적화.

- **구조화된 정보**: 가이드 섹션을 FAQ 형태로 구성. AI가 발췌하기 좋은 질문+답변 구조
- **신뢰 시그널**: 팩트 기반 구체적 설명 ("브라우저에서 처리, 서버 전송 없음")
- **Schema Markup**: SoftwareApplication, HowTo, FAQPage
- **비교 콘텐츠**: 블로그에 도구 비교 글 → AI 참조율 증가

## 6. 개발 프로세스 & 코드 구조

### 6.1 코드 구조 (Next.js App Router)

플랫폼 공통 영역과 서비스별 영역을 폴더로 분리하여, 새 서비스 추가 시 기존 코드를 건드리지 않는 구조.

```
src/
  app/
    page.tsx                        ← 메인 페이지
    blog/                           ← 블로그
    tools/
      image/
        compress/page.tsx           ← 이미지 압축
        resize/page.tsx             ← 리사이즈
        convert/page.tsx            ← 포맷 변환
        heic-to-jpg/page.tsx        ← HEIC→JPG 별도 SEO 랜딩
        crop/page.tsx               ← 크롭
      text/                         ← (향후) 텍스트 도구
        counter/page.tsx
  components/
    common/                         ← Header, Footer, ToolLayout, AdSlot
    image/                          ← 이미지 도구 전용 컴포넌트
    text/                           ← (향후) 텍스트 도구 전용
  lib/
    common/                         ← 공통 유틸 (fileUtils 등)
    image/                          ← 이미지 처리 로직
    text/                           ← (향후) 텍스트 처리 로직
```

**새 서비스 추가 시**: `tools/[service]` + `components/[service]` + `lib/[service]`만 새로 생성. 기존 코드 수정 불필요.

### 6.2 Git 브랜치 전략

| 브랜치 | 역할 |
|--------|------|
| `main` | 실제 서비스 중인 코드. floor05.com에 자동 배포 |
| `feature/[service]` | 새 서비스 개발용. Vercel 프리뷰 URL 자동 생성 |

#### 서비스 추가 워크플로우 (두 번째 서비스부터)

| # | 단계 | 상세 |
|---|------|------|
| 1 | 브랜치 생성 | `feature/text-tools` 브랜치를 main에서 분기 |
| 2 | 개발 | `tools/text`, `components/text`, `lib/text` 폴더 생성 및 개발 |
| 3 | 프리뷰 테스트 | Vercel 프리뷰 URL에서 실제 환경 테스트 |
| 4 | main 머지 | 문제없으면 main에 머지 → 자동 배포 |

이 방식으로 새 서비스 개발 중에 기존 서비스가 영향받지 않는다.

### 6.3 첫 번째 개발 (웹사이트 + 이미지 도구)

첫 번째 서비스는 웹사이트 기반과 함께 한 스프린트로 개발한다. 빈 웹사이트를 먼저 배포하는 것은 의미가 없으므로 함께 진행하되, 커밋 단위로 Phase를 구분한다.

| Phase | 범위 | 상세 |
|-------|------|------|
| Phase 1 | 웹사이트 기반 | 메인 페이지(Hero+도구카드), 헤더, 푸터, ToolLayout, 공통 컴포넌트 |
| Phase 2 | 이미지 도구 | 압축, 리사이즈, 포맷변환, 크롭, 일괄처리, 워크플로우 |
| Phase 3 | SEO/블로그/배포 | 메타태그, Schema, 사이트맵, 블로그, Analytics, 배포 |

## 7. 경쟁사 분석

### 7.1 경쟁사 요약

| 서비스 | 처리 | 한국어 | 요금 | 월 PV | 강점 | 위협도 |
|--------|------|--------|------|-------|------|--------|
| iLoveIMG | 서버 | △번역 | 무료+€4 | 3억 | 도구 15+개 | ★★★★★ |
| TinyPNG | 서버 | ✕영어 | 무료+$25 | 5천만 | 압축품질 최고 | ★★★ |
| Squoosh | 브라우저 | ✕영어 | 완전무료 | 200만 | B/A UI | ★★★★ |
| 비볼디 | 서버 | ◎네이티브 | 무료+유료 | 50만 | 한국어+고품질 | ★★★★ |
| 이미지프레소 | 서버 | ◎네이티브 | 무료+기업 | 30만 | B2B+한국SEO | ★★★ |
| Adobe Express | 서버 | ○지원 | 무료+유료 | 1억 | AI+브랜드 | ★★ |

### 7.2 시장 빈틈 6개

#### Gap 1: 브라우저 처리 + 일괄 처리 = 부재 [영향도 5/5]
Squoosh는 1장씩, iLoveIMG는 서버 필수. 동시 제공 서비스 부재.

#### Gap 2: 한국어 네이티브 + 브라우저 처리 = 부재 [영향도 5/5]
비볼디/이미지프레소=한국어+서버, Squoosh=브라우저+영어. 한국어+브라우저 조합 부재.

#### Gap 3: 목표 용량 설정 = 거의 없음 [영향도 4/5]
"200KB 이하로" 목표 지정. ResizePixel만 제공(서버). 증명사진/지원서 한국 수요.

#### Gap 4: SNS 프리셋 (한국 포함) = 거의 없음 [영향도 4/5]
네이버, 카카오, 당근 포함 한국 SNS 프리셋 제공 도구 부재.

#### Gap 5: HEIC→JPG 한국어 도구 부족 [영향도 4/5]
아이폰 보급 상승. '아이폰 사진 변환' 월 2만+. 브라우저 한국어 도구 부재.

#### Gap 6: 도구 간 워크플로우 = 약함 [영향도 3/5]
대부분 기능 분리. 압축→리사이즈 시 재업로드 필요.

### 7.3 포지셔닝

- 워크플로우 CTA: 압축 완료 → "크기도 조절할까요?"

- 일괄 변환 + ZIP 다운로드
- ⭐ HEIC→JPG 변환 (아이폰 사진):
  - heic2any 라이브러리로 브라우저 기반 HEIC 디코딩

## 10. 에러 & 엣지 케이스

### 10.1 파일 제한

| 항목 | 데스크톱 | 모바일 |
|------|----------|--------|
| 최대 파일 크기 | 50MB/파일 | 50MB/파일 |
| 일괄 최대 장수 | 10장 | 5장 |

### 10.2 브라우저 대응

- WebP 미지원: 해당 옵션 비활성화 + "이 브라우저에서는 WebP를 지원하지 않습니다" 안내
- HEIC heic2any: 변환 페이지 진입 시에만 로드 (초기 로딩 최적화)
- 모바일: 5장 제한으로 메모리 부족 방지

### 10.3 UX 안내 메시지

- 미지원 포맷: "지원하지 않는 파일 형식입니다. JPG, PNG, WebP, HEIC 파일을 사용해주세요."
- 크기 초과: "50MB 이하 파일만 처리할 수 있습니다."
- 처리 중 이탈: "작업이 완료되지 않았습니다. 페이지를 떠나시겠습니까?"
- 처리 실패: "이 파일은 처리할 수 없습니다. 다른 파일을 시도해주세요."
- 모든 에러는 기술 용어 없이 친절한 한국어로
